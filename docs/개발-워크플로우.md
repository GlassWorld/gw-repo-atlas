# 개발 워크플로우

## 기본 순서

1. 변경 요청이 어느 계층에 속하는지 확인합니다.
2. 관련 파일을 먼저 읽고 영향 범위를 좁힙니다.
3. 구현은 계층 책임을 지키면서 진행합니다.
4. 변경 후 검증을 수행합니다.
5. 필요한 문서를 함께 갱신합니다.

## 계층 원칙

### API

- 요청과 응답 처리만 담당합니다.
- 실제 로직은 `server/services/`로 위임합니다.

### 서비스

- 저장소 분석, Git 호출, OpenAI 호출, Q&A 흐름을 담당합니다.

### 데이터베이스

- Prisma를 통해 접근합니다.
- 스키마 변경 시 마이그레이션 여부를 함께 판단합니다.
- DB 작업 범위는 `repo_atlas` 스키마로 한정합니다.
- 다른 스키마의 테이블, 뷰, 함수, 시퀀스는 수정하지 않습니다.
- 테이블을 추가하거나 변경할 때는 테이블별 한글 코멘트를 필수로 작성합니다.
- 컬럼을 추가하거나 변경할 때는 컬럼별 한글 코멘트를 필수로 작성합니다.
- Prisma 스키마 설명과 DB의 `COMMENT ON TABLE` 내용을 함께 관리합니다.
- Prisma 필드 설명과 DB의 `COMMENT ON COLUMN` 내용을 함께 관리합니다.

### UI

- 페이지는 화면 조합 중심으로 작성합니다.
- 공통 UI는 `components/`로 분리합니다.

## 검증 기준

- 구조 변경이나 타입 변경 시 `npm run build`
- DB 스키마 변경 시 `npx prisma migrate dev`
- 환경 변수 변경 시 `.env.example`과 관련 문서 갱신
