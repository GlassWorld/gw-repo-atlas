# 프로젝트 헬스 스코어링

RepoAtlas의 헬스 스코어는 저장소를 하나의 공통 기준으로만 평가하지 않는다. 먼저 프로젝트 종류를 감지한 뒤, 감지된 종류에 맞는 파일과 설정을 대상으로 점수를 계산하도록 OpenAI 요청을 구성한다.

## 처리 흐름

1. Git 저장소를 clone하고 `git ls-files` 기반 파일 목록을 수집한다.
2. 슬롯 기반 규칙으로 README, 매니페스트, 엔트리포인트, 설정 파일, 코드 파일 스니펫을 고른다.
3. 파일트리와 스니펫 경로를 기준으로 프로젝트 종류를 감지한다.
4. 감지된 프로젝트 종류에 맞는 헬스 스코어 기준을 만든다.
5. OpenAI에는 `detectedProject`와 `healthScoreRules`를 함께 전달한다.
6. 응답 JSON의 `healthScore`에는 점수뿐 아니라 감지된 프로젝트 종류와 실제 적용 기준도 저장한다.

## 감지 대상

현재 감지하는 프로젝트 종류는 다음과 같다.

- `nuxt`: `nuxt.config.*`, `app.vue`, `pages/**/*.vue`
- `vue-vite`: `vite.config.*`와 Vue 파일
- `next`: `next.config.*`
- `node`: `package.json`
- `java-spring-gradle`: Gradle 파일과 Spring Boot 신호
- `java-spring-maven`: `pom.xml`과 Spring Boot 신호
- `java-gradle`: Gradle 파일
- `java-maven`: `pom.xml`
- `python`: `pyproject.toml`, `requirements.txt`, `setup.py`
- `go`: `go.mod`
- `rust`: `Cargo.toml`
- `unknown`: 판단 근거가 부족한 경우

## 스코어링 항목

저장 형식과 UI 비교를 단순하게 유지하기 위해 최상위 점수 항목은 유지한다.

- 문서화: 25%
- 테스트 하네스: 25%
- CI/CD: 20%
- AI 코딩 설정: 15%
- 코드 품질: 15%

다만 각 항목에서 보는 대상은 프로젝트 종류별로 달라진다.

## 예시

Nuxt 프로젝트의 테스트 하네스는 `package.json` scripts, `vitest.config.*`, `jest.config.*`, `playwright.config.*`, `cypress.json`, `tests/`를 중심으로 본다.

Java Gradle 프로젝트의 테스트 하네스는 `build.gradle(.kts)`의 `testImplementation`, JUnit, Testcontainers, `src/test`, Gradle `test` 태스크를 중심으로 본다.

Python 프로젝트의 테스트 하네스는 `pytest.ini`, `pyproject.toml`의 pytest 설정, `tox.ini`, `noxfile.py`, `tests/`를 중심으로 본다.

## 응답 구조

`healthScore`에는 다음 정보가 포함된다.

- `projectKind`: 감지된 프로젝트 종류, 신뢰도, 감지 근거
- `documentation`, `testHarness`, `cicd`, `vibeCoding`, `codeQuality`: 각 항목의 점수와 한 줄 근거
- `total`: 가중 평균 총점
- `appliedRules`: 감지된 프로젝트 종류에 따라 실제 적용한 점검 기준
- `flags`: 주의사항
- `suggestions`: 개선 제안

## 주의사항

프로젝트 종류 감지는 파일트리와 선택된 스니펫만 사용한다. 따라서 매니페스트나 설정 파일이 스니펫으로 선택되지 않았거나, 파일명이 일반적이지 않은 프로젝트는 `unknown` 또는 더 넓은 범주로 감지될 수 있다.

## 표시 기준

화면에서는 점수 구간에 따라 색상과 이모지를 함께 표시한다.

- 80점 이상: 좋음
- 50점 이상 80점 미만: 보통
- 50점 미만: 주의

이 표시 기준은 `utils/analysis-presentation.ts`에서 관리한다.
